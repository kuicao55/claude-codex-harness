# Harness Init — Project Context Generation

Generate a comprehensive `status/PROJECT.md` by reading the entire codebase.

**Announce at start:** "I'm using the harness-init skill to analyze this codebase and generate the project context document."

## When to Invoke

- First time using super-harness in a project (before brainstorm)
- User manually runs `/super-harness:init`

## The Process

### Step 1: Run Pre-flight Check

```bash
scripts/harness-preflight
```

Ensure directories exist before generating context.

### Step 2: Detect Project Type

Run appropriate commands to detect:

```bash
# Language/framework detection
ls *.py *.js *.ts *.go *.rs *.java 2>/dev/null | head -20

# Package managers
ls package.json pyproject.toml go.mod Cargo.toml pom.xml 2>/dev/null

# Test frameworks
ls tests/ __tests__/ spec/ 2>/dev/null

# Build tools
ls Makefile CMakeLists.txt 2>/dev/null
```

### Step 3: Read Key Files

Read in this order:

1. README.md (if exists) — project overview
2. package.json / pyproject.toml / go.mod / Cargo.toml / pom.xml — dependencies and scripts
3. Key source files — explore src/ or lib/ or app/ directories
4. Existing test files — understand testing patterns
5. Any existing docs/architecture*.md — capture architectural decisions
6. `git remote -v` or `.git/config` — project name from remote

### Step 4: Explore Functional Modules

For each top-level source directory:
1. List files
2. Read main entry points (index.js, main.py, etc.)
3. Identify what each module does in 1 line
4. Map module → source files

### Step 5: Detect Architectural Decisions

From reading source files and docs, identify:
- Key patterns used (MVC, service layer, repository, etc.)
- Data storage approach (database type, ORM, etc.)
- API style (REST, GraphQL, etc.)
- Any significant design choices

### Step 6: Check Harness State

If `status/claude-progress.json` exists:
- Read current milestone and progress
- Read active branch from `git branch --show-current`
- List existing milestones

### Step 7: Generate PROJECT.md

Write to `status/PROJECT.md`:

```markdown
# Project Context

> Auto-generated by super-harness. Read this first in every session.

## Project Identity

**Project Name:** <from git remote or directory name>
**Harness Version:** 3.2.0
**Generated:** <YYYY-MM-DD>
**Last Updated:** <YYYY-MM-DD>

## Tech Stack

**Languages:** <e.g., Python 3.11, TypeScript>
**Frameworks:** <e.g., FastAPI, React>
**Test Framework:** <e.g., pytest, vitest>
**Package Manager:** <e.g., pip, npm, cargo>

## Functional Modules

| Module | Purpose | Source Location |
|--------|---------|-----------------|
| <name> | <1-line purpose> | <path> |
| ... | ... | ... |

## Key Architectural Decisions

- <decision 1>: <rationale>
- <decision 2>: <rationale>

## Project Structure

```
<project root tree - top 2-3 levels, key directories only>
```

## Completed Milestones

| Milestone | Title | Tasks | Completed |
|-----------|-------|-------|----------|
| milestone-1 | Core API | 5 | 2026-04-10 |
| milestone-2 | User Auth | 3 | — |

## Harness Reference

**Commands:** `/super-harness:brainstorm`, `/super-harness:plan`, `/super-harness:execute`, `/super-harness:resume`, `/super-harness:status`, `/super-harness:init`, `/super-harness:handoff`

**Key Files:**
- `status/claude-progress.json` — Milestone tracker
- `status/PROJECT.md` — This file
- `docs/harness/specs/` — Design specs
- `docs/harness/plans/` — Implementation plans
- `docs/harness/handoffs/` — Session handoffs
- `logs/activity-YYYY-MM-DD.jsonl` — Activity logs
```

### Step 8: Commit

```bash
git add status/PROJECT.md && git commit -m "harness: generate project context"
```
